# 产品需求文档（PRD）

## 产品名称：Claude GitLab 智能助手

### 版本：v1.0

### 最后更新：2026-04-04

***

## 1. 背景与目标

### 1.1 业务背景

开发团队在使用 GitLab 进行协作时，面临以下痛点：

- **重复性工作**：大量简单的代码审查、问题解答、Issue 转 MR 等操作消耗开发者时间。
- **流程割裂**：开发者需要在 GitLab 和 AI 工具（如 Claude）之间来回切换，复制粘贴代码和日志。
- **审查标准不统一**：不同审查者关注点不同，容易遗漏潜在问题。
- **Issue 到 MR 的转化成本高**：从 Issue 描述到实际代码修改、分支创建、MR 提交，需要多个手动步骤。

### 1.2 产品目标

打造一个深度集成于 GitLab 的 AI 助手，使团队能够通过自然语言交互完成以下核心任务：

- 在 Issue / MR 中 **@claude** 进行技术问答、代码解释、任务执行。
- 自动对非 Claude 创建的 MR 进行**代码审查**，输出结构化意见。
- 根据 Issue 描述**自动生成代码变更并创建 MR**。

### 1.3 成功指标（KPI）

| 指标                 | 目标值              |
| :----------------- | :--------------- |
| 人工代码审查耗时降低         | ≥ 30%            |
| Issue → MR 平均创建时间  | 从 2 小时降至 ≤ 10 分钟 |
| Claude 助手满意度（用户调研） | ≥ 4.5 / 5        |
| 自动审查建议采纳率          | ≥ 70%            |
| Webhook 响应时间（P95）  | ≤ 200ms          |
| 服务可用性              | ≥ 99.5%          |

| 角色            | 说明                  | 主要使用场景                         |
| :------------ | :------------------ | :----------------------------- |
| **开发者**       | 日常提交代码、参与 MR 审查     | 在 Issue 中请求 Claude 协助；接收自动审查意见 |

## 3. 功能需求

### 3.1 评论交互：`@claude` 智能问答

**功能描述**\
用户在 Issue 或 Merge Request 的评论区输入 `@claude` 后跟任意自然语言指令，Claude 理解指令并返回回答。

**触发规则**

- 格式：`@claude <指令>`（大小写不敏感，前后可有空格）
- 指令长度：≤ 2000 字符
- 响应位置：在原对话下方新建评论，内容以 `🤖 Claude 回复：` 开头

**能力范围（支持）**

- 解释代码片段（用户可粘贴代码或引用文件路径）
- 分析错误日志
- 生成单元测试样例
- 建议代码优化方案
- 回答 GitLab CI/CD 配置问题
- 读取工作空间内的文件（需显式授权）

**能力范围（不支持）**

- 直接修改代码并提交（需使用 `/create-mr` 命令，见 3.3）
- 删除仓库或分支
- 修改项目设置或成员权限
- 执行破坏性命令（如 `rm -rf`）

**边界处理**

| 场景                | 响应                              |
| :---------------- | :------------------------------ |
| 无指令（只有 `@claude`） | “请提供具体指令，例如 `@claude 解释一下这个函数`” |
| 指令包含敏感操作          | “该操作不被允许，请人工执行”                 |
| Claude 响应超时（>60秒） | “处理超时，请简化指令或稍后重试”               |

***

### 3.2 自动代码审查

**功能描述**\
当有人创建或重新打开一个 Merge Request 时，如果 MR 的作者不是 Claude 助手本人，则 Claude 自动获取代码变更并发表审查意见。

**触发事件**

- MR 状态变为 `opened` 或 `reopened`（不响应 `update` 事件，避免重复审查）

**审查内容**

- 获取 MR 的所有变更文件及 diff
- 审查维度：逻辑错误、性能问题、安全隐患、代码风格、可读性、测试覆盖建议

**输出格式**

- 在 MR 下新建评论，内容以 `🔍 Claude 自动代码审查：` 开头
- 按文件分点列出问题，每条标注严重程度：
  - 🔴 **阻塞**：必须修复才能合并
  - 🟡 **建议**：推荐修改
  - 🟢 **优化**：可选改进
- 无问题时回复：“✅ 未发现明显问题，代码质量良好”

**限制与配置**

| 配置项                     | 默认值    | 说明                       |
| :---------------------- | :----- | :----------------------- |
| `auto_review_enabled`   | true   | 项目级开关                    |
| `max_review_files`      | 20     | 超过此数量不审查，提示“MR 过大，请人工审查” |
| `max_review_diff_chars` | 100000 | diff 字符数上限               |
| `exclude_paths`         | `[]`   | 跳过审查的文件路径模式（如 `*.lock`）  |

**隐私与安全**

- 审查过程中代码变更仅临时传递给 Claude API，不持久化存储
- 不会审查来自 Claude 自己的 MR

### 3.3 基于 Issue 创建 Merge Request

**功能描述**\
用户在 Issue 中评论 `@claude /create-mr`（或包含“创建 MR”意图的自然语言），Claude 会分析 Issue 内容，生成代码变更，提交到新分支，并通过 GitLab API 创建 MR，最后将 MR 链接回复到 Issue。

#### 3.3.1 触发方式

支持两种方式（优先匹配斜杠命令）：

- **精确命令**：`@claude /create-mr`
- **自然语言**：`@claude 请根据这个 Issue 创建一个 MR`、`@claude 帮我实现这个功能并提 MR`

#### 3.3.2 执行流程

| 步骤 | 动作                                  | 负责组件              |
| :- | :---------------------------------- | :---------------- |
| 1  | 校验 Issue 状态（必须为 `opened`，未关联已关闭 MR） | Webhook 服务        |
| 2  | 获取 Issue 标题、描述、所有评论（作为上下文）          | GitLab API        |
| 3  | 准备独立工作空间（克隆仓库，切到默认分支）               | Workspace Manager |
| 4  | 调用 Claude CLI，提供结构化提示词              | Claude CLI        |
| 5  | Claude 生成代码变更、运行测试、提交、推送分支          | Claude CLI        |
| 6  | 解析 CLI 输出，获取新分支名                    | Webhook 服务        |
| 7  | 调用 GitLab API 创建 MR                 | GitLab API        |
| 8  | 在 Issue 下回复 MR 链接                   | GitLab API        |

#### 3.3.3 Claude CLI 提示词结构（模板）

```text
你是一个资深开发者。当前工作目录是项目仓库，默认分支为 {default_branch}。

目标：根据以下 Issue 内容，实现需求并创建一个 Merge Request。

Issue 内容：
[标题]：{title}
[描述]：{description}
[评论]：{comments}

请按以下步骤操作：
1. 分析需求，确定需要修改/新增的文件。
2. 使用 Edit/Write 工具修改代码。
3. 如果存在测试命令（如 npm test），运行并确保通过；若失败，尝试修复。
4. 使用 Bash 工具执行：git add . && git commit -m "{commit_message} #issue_{iid}"
5. 创建新分支名：claude/issue-{iid}-{short_desc}（short_desc 由你生成，不超过30字符）
6. 推送分支：git push origin HEAD:refs/heads/{branch_name}

完成后，请输出最终推送的分支名称（JSON 格式：{"branch": "..."}）。
```

#### 3.3.4 分支与 MR 规范

| 项目      | 规范                                                     |
| :------ | :----------------------------------------------------- |
| 分支命名    | `claude/issue-<issue_iid>-<short_desc>`                |
| 提交信息    | `<动词> <内容> #<issue_iid>`，如 `Fix login button bug #123` |
| MR 标题   | `[Claude] <Issue 标题>`                                  |
| MR 描述模板 | 见下表                                                    |

**MR 描述模板**：

```markdown
此 MR 由 Claude 自动生成，基于 Issue #{iid}。

**Claude 的变更说明**：
(Claude 生成的摘要)

**测试情况**：
(若运行了测试，输出结果)

**人工审阅提醒**：请确认变更符合预期后再合并。
```

#### 3.3.5 失败处理

| 失败场景                  | 用户提示                                |
| :-------------------- | :---------------------------------- |
| Issue 描述不清晰           | “Issue 描述不够清晰，请补充具体修改点后再试。”         |
| 仓库无法访问（clone/push 失败） | “无法访问代码仓库，请检查 Token 权限或网络。”         |
| Claude CLI 执行超时（>5分钟） | “任务执行超时，请尝试简化需求或手动实现。”              |
| 代码修改导致测试失败且无法修复       | “自动修复失败，以下是生成的 diff：...(附上) 请人工介入。” |
| 分支名冲突                 | 自动在分支名后添加时间戳重试。                     |
| 创建 MR API 失败          | 返回 API 错误信息，并提供手动创建指引。              |

#### 3.3.6 安全与限制

| 限制项    | 说明                                                        |
| :----- | :-------------------------------------------------------- |
| 项目白名单  | 仅对显式启用的项目开放（`create_mr_enabled=true`）                     |
| 路径黑名单  | Claude 不得修改 `.gitlab-ci.yml`、`Dockerfile`、`config/` 等关键文件 |
| 强制人工复核 | Claude 创建的 MR 必须经过至少一位维护者批准才能合并                           |
| 并发限制   | 同一项目同时只能有一个“创建 MR”任务运行                                    |
| 每日配额   | 每个项目每天最多创建 5 个 Claude MR（可配置）                             |

## 4. 非功能需求

### 4.1 性能要求

| 指标                     | 目标值     |
| :--------------------- | :------ |
| Webhook 端点响应时间（P95）    | ≤ 200ms |
| `@claude` 问答端到端延迟（P95） | ≤ 15 秒  |
| 自动代码审查端到端延迟（P95）       | ≤ 30 秒  |
| 创建 MR 任务端到端延迟（P90）     | ≤ 5 分钟  |

### 4.2 可靠性要求

- 服务可用性 ≥ 99.5%（按月统计）
- 任务失败重试：网络瞬时错误自动重试最多 3 次（指数退避）
- 所有关键操作（创建 MR、发表评论、清理工作空间）均记录结构化日志
- 提供健康检查端点（`/health`），供负载均衡和监控使用

### 4.3 安全性要求

| 安全项          | 要求                                                                               |
| :----------- | :------------------------------------------------------------------------------- |
| Webhook 签名验证 | 所有请求必须携带 `X-Gitlab-Token` 或 `X-Hub-Signature-256`，验证失败返回 401                     |
| 敏感信息存储       | GitLab Token 和 Anthropic API Key 必须加密存储（AES-256）                                 |
| 运行隔离         | Claude CLI 运行在 Docker 容器中，限制 CPU（0.5核）、内存（1GB）、网络（仅允许出站到 GitLab 和 Anthropic API） |
| 命令白名单        | 禁止 Claude 执行危险命令（`rm -rf /`、`sudo`、`curl` 外网等），通过容器策略或 CLI 配置限制                  |
| 分支保护         | 禁止 force push 到 `claude/*` 分支（通过 GitLab 分支规则）                                    |

### 4.4 可扩展性要求

- Webhook 接收器无状态，支持水平扩展（多实例）
- 任务队列使用 Redis，保证任务不丢失且可重放
- 支持多 GitLab 实例（SaaS 或自托管）

### 4.5 可观测性要求

- 暴露 Prometheus 指标：
  - `claude_webhook_requests_total`（按事件类型）
  - `claude_task_duration_seconds`（按任务类型）
  - `claude_task_success_total` / `failed_total`
  - `claude_workspace_count`（当前工作空间数）
- 错误告警：连续 5 分钟错误率 >10% 时，发送告警到钉钉/Slack

## 5. 工作空间管理

### 5.1 独立工作空间定义

每个任务（以 Issue IID 或 MR IID 为标识）拥有**完全隔离的工作空间**，包括：

- 独立的代码仓库副本（浅克隆 `--depth=1`）
- 独立的临时文件目录
- 独立的 Claude CLI 执行环境（Docker 容器或进程命名空间）

**工作空间标识符**：

- Issue 交互：`workspace-issue-<project_id>-<issue_iid>`
- MR 自动审查：`workspace-mr-<project_id>-<mr_iid>`

### 5.2 生命周期管理

| 阶段        | 行为                                                                | 触发条件                  |
| :-------- | :---------------------------------------------------------------- | :-------------------- |
| **创建**    | 在 `WORKSPACE_ROOT` 下新建目录，`git clone --depth=1` 项目仓库               | 首次收到该 Issue/MR 的任务    |
| **复用**    | 执行 `git fetch && git reset --hard origin/<default_branch>` 恢复干净状态 | 同一 Issue/MR 再次触发任务    |
| **关闭时清理** | 递归删除工作空间目录                                                        | Issue 被关闭 或 MR 被合并/关闭 |
| **超时清理**  | 删除空闲超过 `WORKSPACE_TTL_HOURS` 的工作空间                                | 后台定时任务（每小时扫描一次）       |

### 5.3 并发与锁机制

- 同一个工作空间同时只能有一个任务执行（使用 Redis 分布式锁或文件锁 `flock`）
- 不同工作空间完全隔离，可并发执行
- 全局最大并发工作空间数可配置（默认 10）

### 5.4 资源配额

| 资源           | 限制                               |
| :----------- | :------------------------------- |
| 每个工作空间最大磁盘占用 | 500MB                            |
| 全局最大工作空间数    | 50（可配置 `MAX_WORKSPACES`）         |
| 单个任务最长执行时间   | 5 分钟（超时强制终止）                     |
| 空闲 TTL       | 24 小时（可配置 `WORKSPACE_TTL_HOURS`） |

### 5.5 清理机制详解

#### 5.5.1 基于状态变更的主动清理

- **监听事件**：
  - `Issue Hook` 中 `action = "close"`
  - `Merge Request Hook` 中 `action` 为 `close` 或 `merge`
- **处理逻辑**：
  1. 根据 `project_id` 和 `iid` 构造成工作空间路径
  2. 若目录存在，递归删除
  3. 记录日志
- **安全措施**：
  - 仅删除以 `workspace-issue-` 或 `workspace-mr-` 开头的目录
  - 删除前检查是否有活跃任务锁，若有则延迟 30 秒重试

#### 5.5.2 超时清理（兜底机制）

- 后台定时任务（cron 每小时执行）
- 扫描 `WORKSPACE_ROOT` 下所有工作空间，删除最后修改时间超过 `WORKSPACE_TTL_HOURS` 的目录
- 保证即使关闭事件丢失，空间也不会永久占用

#### 5.5.3 清理优先级

| 优先级 | 清理方式  | 说明          |
| :-- | :---- | :---------- |
| 高   | 关闭时清理 | 正常流程结束，立即释放 |
| 低   | 超时清理  | 兜底，处理异常残留   |

## 7. 用户交互流程

### 7.1 典型场景一：开发者请求 Claude 修复一个 bug

1. **开发者**：在 Issue #123 中描述 bug，然后评论 `@claude 请修复这个 bug，并创建 MR`。
2. **Claude 助手**（自动）：
   - 回复“正在处理，请稍候...”。
   - 分析 Issue，生成代码补丁，推送到分支 `claude/issue-123-fix-null-pointer`。
   - 创建 MR #456，标题为 `[Claude] Fix null pointer exception in login #123`。
   - 在 Issue #123 中回复：“已创建 MR #456，请审阅。”
3. **开发者**：查看 MR #456，确认变更正确，批准并合并。
4. **Claude 助手**：不主动关闭 Issue，由开发者手动关闭。当 Issue 关闭时，工作空间自动清理。

### 7.2 典型场景二：自动代码审查

1. **开发者 A**：创建 MR #789，修改了 `auth.py`。
2. **Claude 助手**（自动）：
   - 检测到非 Claude 创建的 MR，获取 diff。
   - 发表审查评论：“🔴 阻塞：第 42 行未对用户输入做校验，可能导致 SQL 注入。建议使用参数化查询。”
3. **开发者 A**：根据建议修改代码，push 新 commit。
4. **Claude 助手**：不重复审查（仅对 open/reopen 触发）。若需再次审查，开发者可手动 `@claude /review`（扩展功能，v1.1）。

### 7.3 典型场景三：纯问答

1. **开发者**：在 MR #101 的评论区 `@claude 解释一下这个错误日志的含义`。
2. **Claude 助手**：分析日志，回复解释和建议。
3. **开发者**：继续追问 `@claude 那如何修复呢？`（同一 MR 内，复用工作空间，Claude 能记住对话上下文）。

### 9.1 风险清单

| 风险                  | 概率 | 影响 | 缓解措施                                                    |

## 10. 验收标准

### 10.1 功能验收

- **评论问答**：在测试 Issue 中 `@claude 1+1等于几`，10 秒内收到正确答案。
- **自动代码审查**：创建测试 MR（非 claude-bot 创建），30 秒内收到审查评论，评论包含严重程度标签。
- **创建 MR**：在 Issue 中 `@claude /create-mr`，成功创建 MR 并回复链接，且 MR 包含符合 Issue 描述的代码变更。
- **工作空间隔离**：并发触发两个不同 Issue 的 `@claude`，它们的工作空间目录不同，互不影响。
- **工作空间复用**：同一 Issue 的第二次 `@claude` 复用了之前的工作空间，且仓库状态已重置为最新。
- **关闭时清理**：关闭 Issue 或合并 MR 后，对应的工作空间目录在 10 秒内被删除。
- **超时清理**：闲置超过 24 小时的工作空间被自动删除。

### 10.2 性能验收

- Webhook 响应时间 ≤ 200ms（95% 请求）。
- 自动审查任务完成时间 ≤ 30 秒（95% 任务）。
- 创建 MR 任务完成时间 ≤ 5 分钟（90% 任务）。

### 10.3 安全验收

- 未提供正确 Secret Token 的 Webhook 请求返回 401。
- Claude CLI 在 Docker 容器中无法删除宿主机文件（通过容器限制验证）。
- Claude 创建的 MR 包含强制人工批准规则（GitLab 分支保护设置）。
